01 完整工作流

6步 Pipeline（一个可迭代的循环）

用户从"我关心某个品类"走到"我知道该打哪个格子、机会在哪"，经过6个步骤。这不是一条单向流水线，而是一个循环——诊断后发现维度不对、归一太粗，可随时回到任意上游步骤重绘，直到网格"可用"为止。每个环节都有人机交互的设计——在用户本来就要做决策的瞬间，系统主动提醒并顺手捕获偏好，所有策略碎片统一沉淀入ERP。

进入方式（前置）：从插件中心选择「品类网格拆解」→ 先确定 部门（单选） + 品类（多选） + 时间维度 → 再进入下面的 Step 1 货盘圈选。

05 Step 4

网格绘制与调优

按选定维度和归一后的属性值交叉生成网格，商品归格，标记置信度。初绘后系统自动预检并给出优化建议，用户通过结构化控件完成调优——合并碎片、归组属性值、标注重点，每步有明确引导。

行/列维度可在网格上快速切换：价格带 / 品牌类型 / 功能类型，也可选 品类（需选一级 / 二级 / 三级）；填充指标可切换 增速 / 市占 / GMV。

初绘后自动预检

检查项阈值处理
空格率>30%提示维度过细，建议合并
最大格占比单格>50%商品提示维度值过粗，建议拆分
碎片格单格<2个商品提示该区域可合并

引导式调优交互

不用对话，不用记指令。系统主动推送建议，用户通过结构化控件操作——点击、拖拽、勾选，每步有明确引导。

设计原则：调优只针对当前已画出来的网格做优化——合并碎片、归组属性值、标注重点。不再引入新维度或新数据，避免"调着调着回到上游"。

06 Step 5 · 属性提取与归一（画格后）

属性提取与归一

本环节在"网格画出来之后"执行（位于网格绘制与调优之后、诊断之前）。先按选定维度把格子画出来，再逐格做两件事：① 提取校验——检查商品挂的属性是否正确，不正确的让用户修正；② 归一——将过细的属性值合并为有业务含义的统一值。这是网格质量的根基——脏属性值=废网格。

场景一：属性校验与修正（画完网格后）

网格画完后，系统逐格检查商品的属性挂载情况。模型为每个商品的属性挂载计算置信度分数，置信度较低的标记为待确认，用户可直接修正。

属性校验 — 发现 3 条待确认
置信度 62% 商品"海尔 BCD-180"当前挂载：使用场景=家用
模型推断：标题含"租房小冰箱"，更可能属于"租房"场景，请确认
修正为 租房 保持 家用 自填：____
置信度 35% 商品"容声 BC-98"缺失：保鲜技术
商品描述含"净味保鲜"，建议补充"净味"
补充 净味 跳过
🤖 Agent 检测到多次相同修正
你已3次将"租房"相关商品从"家用"修正为"租房"，要固化为品类归一规则吗？
固化为规则 → 同步至ERP 仅本次使用
沉淀为 normalization_rule → ERP 策略中心

场景二：归一（合并网格时触发）

当用户在调优环节合并网格时，系统通过模型意图识别判断：用户是否因为属性值画得太细才做合并？如果是，则主动询问是否将细粒度属性归一。

归一触发机制：用户合并"珊瑚粉格"+"樱花粉格"+"蜜桃粉格" → 模型识别意图"这些颜色属性值太细了" → 询问用户是否归一为"粉色" → 归一后，后续提取只会出现"粉色"，不会再拆出更细的值。
属性归一修正 — 颜色
当前值 暗黑色 黑色 深黑色 珊瑚粉 樱花粉 蜜桃粉
归一后 黑色 粉色
🤖 Agent 识别到合并意图
你合并了"珊瑚粉/樱花粉/蜜桃粉"三个格子，是因为颜色属性值太细了吗？确认归一后，后续只提取"粉色"。
确认归一 → 固化为规则 仅合并格子，不归一属性
沉淀为 normalization_rule → ERP 策略中心

归一的三种类型

类型问题描述归一方式示例
同义归一同一含义多种表述同义词映射暗黑色/黑色/深黑色 → 黑色
粒度归一属性值粒度过细或过粗区间归并50L/80L/100L → 小型(≤100L)
缺失归一属性值缺失但可推断从商品标题/描述提取商品名含"迷你"→推断容量=小型

08 策略沉淀

品类网格的策略沉淀体系

策略不是独立的业务框架，而是直接绑定在网格拆解过程上的配置包——同一个品类，换了策略，拆出来的维度、归一规则、指标权重、合并建议、诊断结论全都不同。策略沉淀 = 把这些"怎么拆"的知识固化下来，让新人不用重新摸索。

核心原则：策略不是"写"出来的，是"做"出来的。不要让用户额外填策略表，而是在他本来就要做决策的瞬间，顺手捕获。所有捕获的策略碎片最终统一沉淀至 ERP 策略中心，成为组织级可复用资产。

所有策略统一沉淀至 ERP 策略中心。品类负责人在 ERP 端审核碎片、确认入库、管理策略生命周期。跨用户3人以上做了相同操作 → 自动推给品类负责人确认 → 升格为品类策略并写入 ERP。

经验模板库 — 资深经验打包，新人一键复用

沉淀的策略不只用于"下次自动推荐"，还进一步打包为经验模板：把资深运营按品类沉淀好的属性拆分组合 + 指标配置 + 归一/网格调整规则封装成一个品类模板。

新人使用路径：选中对应品类模板后，系统自动走完前四步的默认配置（货盘圈选 / 属性推荐 / 提取归一 / 指标选择），直接跳到 Step 5 查看网格结果，再基于模板做微调。既能复用资深经验，又能让新人在实操里逐步理解每一步的逻辑。

对已有网格：可从任意步骤进入配置，并回看每个环节当初是怎么配的（用了哪些维度、如何归一、挂了哪些指标），支持从任一步重新走流程。

10 架构设计 · 编排

原子能力与 Agent 编排

把上面的 6 步工作流从"流程视角"翻译成"能力视角"：每一步不是黑盒，而是由一批可独立调用、单一职责的原子能力（Tool / Skill / Model / API）拼成。固定的是原子能力，灵活的是编排——Agent 持有对话上下文与策略配置，按用户意图动态组合这些能力，而不是跑一条写死的流水线。

类比：正如 Tool(拉数据) → Skill(诊断) → Skill(策略生成) → Tool(执行) 这样的主干——能力本身保持原子、无状态、可复用；Agent 只决定按什么顺序调、何时回退、何时跳步、何时插入人机确认。

③ Agent 灵活编排（不是固定流水线）

主干只是默认路径。Agent 会依据上下文改变编排——跳步、回退、插分支、按需扩展：

编排能力触发条件Agent 的动作
↺ 循环回退诊断后发现维度选错 / 归一过粗重新调用 维度推荐 → 网格计算，只重算下游，不从头再来
⤼ 跳步（静默跑）命中经验模板 / 模式 B跳过推荐与归一交互：读策略 → 套配置 → 网格计算 → 诊断 直出结果
⑂ 条件分支属性置信度低 / 用户合并格子插入 属性校验 HITL；或调 合并意图识别 决定是否触发归一
＋ 按需扩展用户勾选扩展指标 / 切换策略追加 指标取数 / 份额取数，联动重跑 诊断打标 + 洞察生成

④ 编排定义（示意）

# 主干：Agent 依次编排原子能力 API.商品取数(品类, 条件, TopN) → Model.维度推荐(用户决策 + 站外趋势 + 策略) # HITL: 可增删维度 → Skill.指标映射(策略) + API.指标取数 → Tool.网格计算(维度交叉 → 商品归格 → 指标聚合) → Model.属性校验归一(画格后) # HITL: 低置信度确认 → Skill.诊断打标(阈值规则) → Model.洞察生成 → API.策略沉淀(偏好碎片 → ERP) # 分支 / 循环：Agent 按上下文动态决策 if 命中经验模板: 跳过推荐 & 归一交互, 读策略 → 套配置 → 静默直出 if 属性置信度 < 阈值: 插入 属性校验 HITL 节点 if 用户合并格子: Model.合并意图识别 → 命中则触发 Skill.属性归一 if 诊断后维度不合适: ↺ 回到 维度推荐, 仅重算下游节点 if 用户切换策略: 重跑 诊断打标 + 洞察生成 (数据不变, 只换规则)
编排原则：原子能力保持单一职责、无状态、可组合，谁都能被单独调用或复用到别的插件；编排逻辑集中在 Agent——它持有对话上下文与策略配置，负责决定调用顺序、是否回退 / 跳步、何时插入人机确认。这样既能"模式 A 一步步交互"，也能"模式 B 静默跑模板"，还能在诊断后无缝回到任意上游重绘。
